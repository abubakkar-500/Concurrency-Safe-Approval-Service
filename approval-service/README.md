# Concurrency-Safe Approval Service

Backend assessment service for managing approval requests with correct behavior under concurrent approve/reject operations.

## Tech stack

- Node.js + TypeScript
- NestJS
- PostgreSQL
- Prisma ORM

## Prerequisites

- Node.js 20+
- Docker Desktop (for PostgreSQL)
- npm

The app lives in the `approval-service/` directory. Run all commands below from that folder.

## Setup

1. **Start PostgreSQL**

   ```bash
   docker compose up -d
   ```

2. **Configure environment**

   ```bash
   cp .env.example .env
   ```

3. **Install dependencies**

   ```bash
   npm install
   ```

4. **Apply database migrations**

   ```bash
   npx prisma migrate deploy
   npx prisma generate
   ```

5. **Run the API**

   ```bash
   npm run start:dev
   ```

   The server listens on `http://localhost:3000` by default.

## API

### Create approval request

```bash
curl -X POST http://localhost:3000/approvals \
  -H "Content-Type: application/json" \
  -d "{\"title\":\"Q1 budget\"}"
```

Response (`201`): `id`, `title`, `status` (`PENDING`), `version` (`1`), `createdAt`, `updatedAt`.

### Get approval request

```bash
curl http://localhost:3000/approvals/<id>
```

Response (`200`) or `404` if not found.

### Approve request

```bash
curl -X POST http://localhost:3000/approvals/<id>/approve
```

- `200` when the request was `PENDING` and is now `APPROVED`
- `409` when another client already approved/rejected it
- `404` when the ID does not exist

### Reject request

```bash
curl -X POST http://localhost:3000/approvals/<id>/reject
```

Same status rules as approve, with final status `REJECTED`.

## Tests

**Unit tests** (no database required):

```bash
npm run test
```

**End-to-end tests** (require PostgreSQL running and migrations applied):

```bash
docker compose up -d
npx prisma migrate deploy
npm run test:e2e
```

The e2e suite includes a concurrency test that fires 10 parallel approve requests and asserts exactly one succeeds (`200`) and the rest return `409 Conflict`.

## Design decisions and concurrency approach

### Architecture

The service uses a layered NestJS structure:

- `ApprovalsController` — HTTP routing and validation
- `ApprovalsService` — business rules and state transitions
- `PrismaService` — database access

### Concurrency control

Approve and reject use a **single atomic conditional update** instead of read-modify-write in application code:

```sql
UPDATE approval_requests
SET status = :target, version = version + 1, updated_at = NOW()
WHERE id = :id AND status = 'PENDING';
```

In Prisma this is expressed as `updateMany` with `where: { id, status: PENDING }`. Success is determined by `result.count`:

- `count === 1` — this request won the race; return the updated entity
- `count === 0` — either the ID is missing (`404`) or the row is no longer pending (`409 Conflict`)

This guarantees that when multiple clients approve or reject the same request simultaneously, **only one transition succeeds**. PostgreSQL row-level locking ensures the `UPDATE` statements are serialized for the same row.

The `version` field increments on each successful transition for auditability and aligns with optimistic concurrency patterns, though clients are not required to send a version on approve/reject because correctness is enforced in the database predicate.

### Assumptions

- No authentication (out of assessment scope)
- IDs are UUIDs generated by the database
- Create payload requires only `title`
- Listing endpoints are not required by the brief
